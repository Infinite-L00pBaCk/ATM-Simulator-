1. Project Overview

This project implements a fully-functional ATM (Automated Teller Machine) Simulator in C++ that is directly connected to a Banking System. The ATM reads from and writes to the bank's shared data files, meaning any account created in the Banking System is immediately accessible at the ATM - just like a real-world ATM connected to a bank's central database.

Key Achievement: The ATM Simulator and Banking System share the same accounts.dat and transactions.dat files. Deposits, withdrawals, and transfers made at the ATM are reflected in the Banking System and vice versa.

3. Bank Integration - How It Works

The ATM Simulator connects to the Banking System by reading from and writing to the same data files. This is the core integration mechanism:

3.1 Shared Data Files

Both the Banking System and ATM Simulator use identical pipe-delimited serialization formats:

accounts.dat format:

PNB-1001|PRIYAM PRAKASH|4046|3500.00|1|2026-05-07 19:16:35
PNB-1003|SURYA|1234|5000.00|1|2026-05-14 13:14:30

Fields: AccountNo | Name | PIN | Balance | Active(1/0) | CreationDate

transactions.dat format:

PNB-1001|DEPOSIT|500.00|1500.00|2026-05-07 19:17:05|Cash deposit
PNB-1001|TRANSFER_IN|2000.00|3500.00|2026-05-14 13:11:41|Transfer from PNB-1002

Fields: AccountNo | Type | Amount | BalanceAfter | Timestamp | Description

3.2 Dynamic Account Loading

Every time a user "inserts their card" at the ATM, the system reloads accounts fresh from the bank's data file. This means:

  • New accounts created in the Banking System are instantly available at the ATM
  • Balance changes made in the Banking System are immediately reflected
  • Closed accounts are automatically rejected by the ATM

3.3 Dual-Prefix Support

The ATM supports both account number formats:

  • PNB-xxxx - Accounts created by the Banking System
  • ATM-xxxx - Accounts created via ATM Admin mode
How it works: The ATM constructor parses account numbers by finding the dash (-) separator and extracting the numeric suffix, making it prefix-agnostic. Both PNB and ATM accounts are stored in the same shared file.

4. OOP Concepts Used

4.1 Encapsulation

All class data members are private. Access is controlled through public methods:

class ATMAccount {
private:
    std::string pin;       // Hidden - cannot be read directly
    double balance;        // Modified only via deposit()/withdraw()
public:
    bool verifyPin(const std::string& inputPin) const;
    bool deposit(double amount);   // Controlled write access
    double getBalance() const;     // Controlled read access
};

4.2 Abstraction

The ATMachine class hides the complexity of file I/O, account lookup, daily limits, and receipt generation. The main.cpp simply calls atm.startSession() without knowing internal details.

5.1 ATM Session Flow

5.2 Withdrawal Logic

// Daily limit check (Rs. 25,000 per day)
if (dailyWithdrawnAmount + amount > 25000.00) --> REJECT

// Balance check
if (amount > balance) --> REJECT ("Insufficient balance!")

// Success path
balance -= amount;
dailyWithdrawnAmount += amount;
Log transaction, save accounts, generate receipt

5.3 Fund Transfer Logic

  1. Verify recipient account exists and is active
  2. Check sender != recipient (prevent self-transfer)
  3. Call sender.withdraw(amount)
  4. Call recipient.deposit(amount)
  5. Log TWO transactions: TRANSFER_OUT for sender, TRANSFER_IN for recipient
  6. Save accounts and generate receipt

5.4 PIN Change Logic

  1. Verify current PIN matches
  2. Validate new PIN is exactly 4 digits
  3. Ensure new PIN differs from old PIN
  4. Confirm new PIN (entered twice)
  5. Update PIN, save to file, log transaction

6. Security Features

FeatureImplementation
PIN LockoutAfter 3 consecutive failed PIN entries, the card is blocked for that session
Daily Withdrawal LimitRs. 25,000 maximum per day across all withdrawal types (regular + quick cash)
Session IsolationEach card insertion creates an independent session with fresh daily limits
Inactive Account RejectionClosed/inactive accounts cannot authenticate at the ATM
Input ValidationAll numeric inputs are validated with cin.fail() checks to prevent crashes
Self-Transfer PreventionCannot transfer funds to the same account
